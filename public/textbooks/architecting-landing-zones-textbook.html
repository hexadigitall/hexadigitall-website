Chapter 1: The Blast Radius Theory

In a traditional single-account cloud environment, every resource shares the same logical boundary. This shared boundary is known as the Blast Radius. If credentials are compromised, the entire organization is at risk. Architecting a Landing Zone is primarily an exercise in Blast Radius Reduction.

The Case for Account-Level Segregation

Architecting a Landing Zone is primarily an exercise in Blast Radius Reduction. By moving to a multi-account strategy, we ensure that:

The Glass Ceiling: Service Quotas & API Throttling

Security is the most common reason for a multi-account strategy, but operational physics is often the most urgent. Every cloud account has invisible "hard" and "soft" limits that, once hit, can bring an entire enterprise to a standstill.

1. Service Quotas (Resource Limits)

Cloud providers limit the number of resources (VPCs, EC2 instances, IGWs) you can create per region per account. While some are adjustable, they represent a shared pool. In a single-account setup, a developer in "Dev" could accidentally consume all available Elastic IPs, preventing "Production" from scaling during a traffic spike.

2. API Rate Limits (The Throttling Problem)

Every time you run a terraform plan or a script calls DescribeInstances, you consume API credits. Cloud providers throttle these requests to protect their control plane.

Scenario: If you have 50 microservices in one account all being managed by a CI/CD pipeline, the sheer volume of API calls can trigger RequestLimitExceeded errors. This "noisy neighbor" effect can block critical security patches from deploying.

The Landing Zone Solution

By distributing workloads across multiple accounts, we effectively multiply our API quota and resource limits by the number of accounts in the Organization. This allows the enterprise to grow horizontally without ever hitting the "Glass Ceiling" of a single-account's constraints.

TEACHING NOTES

The Management Plane: AWS vs. Azure

To build a truly cross-platform Landing Zone, an architect must understand how different providers handle account grouping. While AWS uses Organizations, Azure utilizes Management Groups. Both structures allow for policy inheritance, but their implementation logic varies.

1. AWS Organizations (OUs)

In AWS, the Management Account sits at the top. It creates Organizational Units (OUs), which are containers for accounts. A key constraint is that an AWS account can only belong to exactly one OU at a time. Policies (SCPs) applied to an OU are inherited by all child OUs and accounts beneath it.

2. Azure Management Groups

Azure takes a slightly more flexible approach. Management Groups sit above Subscriptions (Azure's equivalent of an AWS account). Unlike AWS, Azure allows for a deep nesting of up to six levels of Management Groups, providing a more granular control over complex corporate hierarchies.

Feature AWS Azure
Root Container Organization Root Root Management Group
Grouping Unit Organizational Unit (OU) Management Group
Individual Unit Account Subscription

Key Strategy: Regardless of the provider, always avoid placing accounts or subscriptions directly in the "Root." Always create at least one layer of management containers to prevent accidental global policy lockouts.

Chapter 3: Least Privilege & Guardrail Theory

In a multi-account environment, the Principle of Least Privilege (PoLP) is no longer just about individual users; it is about the accounts themselves. A Guardrail is a high-level governance policy that provides ongoing oversight of your environment. Unlike traditional firewalls, guardrails are preventive or detective measures that stay in effect even as your infrastructure evolves.

Preventive vs. Detective Guardrails

A specialist architect must balance the rigidity of prevention with the agility of detection:

  • G1. Preventive Guardrails: These stop an action before it happens (e.g., an SCP that prevents any user in a "Production" OU from deleting CloudTrail logs).
  • G2. Detective Guardrails: These identify when a resource is "Out of Compliance" and trigger an alert or automated remediation (e.g., a rule that flags any S3 bucket that is made public).

TEACHING NOTES

The Policy Intersection: SCPs vs. Boundaries vs. IAM

In a Landing Zone, permissions are not just about what a user has; they are about what the environment permits. We use a "defense-in-depth" approach where three distinct policy types intersect to determine the final effective permission.

1. Service Control Policies (SCPs)

SCPs are the "Global Filters." They are applied at the OU or Account level within the Organization. An SCP defines the maximum available permissions for an entire account. Even if a user is given "Full Administrator" rights via IAM, if an SCP denies a service (e.g., "Deny S3"), that user can never access S3.

2. Permission Boundaries

These are the "Safety Nets" for decentralized administration. We use Permission Boundaries to limit the maximum permissions an IAM entity can have. This is essential when you want to allow a developer to create their own IAM roles but want to ensure they cannot create a role with more power than themselves.

3. Identity-Based (IAM) Policies

These are the "Standard Access" grants. They are attached directly to users, groups, or roles. They define exactly what actions the identity can perform on specific resources.

The Evaluation Logic Rule:

Implicit Deny → SCP Check → Boundary Check → IAM Check → Final Decision.
If any layer contains an explicit "Deny," the request is rejected immediately, regardless of what other layers say.

Chapter 4: Inter-account Communication Theory

In Phase I, we isolated our accounts to reduce the Blast Radius. However, isolation without connectivity is an island, not an ecosystem. Inter-account Communication Theory focuses on how to allow disparate VPCs (Virtual Private Clouds) to talk to each other securely without creating a "Spaghetti Mesh" of manual connections.

The Evolution: From Peering to Hub-and-Spoke

As a Landing Zone Specialist, you must understand why we move away from traditional VPC peering for large-scale deployments:

  • T1. Transit Limitations: Standard VPC peering is non-transitive. If VPC A is peered to VPC B, and VPC B is peered to VPC C, VPC A cannot talk to VPC C through B.
  • T2. The "Hub" Concept: By creating a central "Network Account," we deploy a Transit Gateway that acts as a regional router. Every other account (Spoke) simply "attaches" to this hub.
  • T3. Route Table Management: Instead of managing hundreds of peering routes, we centralize routing logic in the Hub, allowing for global traffic inspection.

Transit Gateway vs. VPC Peering at Scale

As a Landing Zone Specialist, you will often be asked: "Why can't we just use Peering?" The answer lies in the complexity of n(n-1)/2. As the number of VPCs grows, the number of peering connections required for a full mesh becomes unmanageable.

1. The Mesh Complexity (VPC Peering)

VPC Peering is a 1-to-1 relationship. If you have 10 VPCs that all need to talk to each other, you need 45 separate peering connections. Each connection requires manual acceptance and individual route table updates. Peering is excellent for high-performance, low-latency links between two specific points, but it does not scale for enterprise "East-West" traffic.

2. The Centralized Router (Transit Gateway)

Transit Gateway (TGW) acts as a regional network hub. You attach your VPCs to the TGW, and it handles the routing between them.

  • • Simplified Routing: One attachment per VPC instead of dozens of peers.
  • • Transitive Routing: VPC A can talk to VPC C through the TGW.
  • • Central Inspection: You can force all traffic through a "Security VPC" for deep packet inspection.

The Architect's Rule of Thumb:

Use VPC Peering for simple, low-latency connections between two accounts within the same region. Use Transit Gateway for any environment involving more than 3-5 VPCs or requiring cross-account centralized security.

Shared VPC vs. Private Link/Endpoints

Selecting the right service delivery model is critical for maintaining a clean IP space and high security. While both patterns allow cross-account communication, they solve for different organizational needs.

1. Shared VPC (Administrative Centralization)

In a Shared VPC model, a central Host Account owns the network infrastructure (VPC, Subnets, Route Tables). It shares specific subnets with Service Accounts.

  • • Best For: Organizations that want a single team to manage all IP addresses and network security groups.
  • • Key Benefit: Simplifies network topology by reducing the number of VPCs that need to be peered or attached to a Hub.

2. Private Link / Interface Endpoints (Zero-Trust)

Private Link allows you to expose a service (backed by a Network Load Balancer) to other VPCs as if it were a local IP address in the consumer's subnet.

  • • Best For: Connecting overlapping IP spaces or providing a service to accounts that should not have full network access to the provider.
  • • Key Benefit: One-way connectivity. The consumer can talk to the service, but the service cannot initiate a connection back to the consumer.

Architect's Decision Matrix:

Does the consumer need to see the entire network? Shared VPC.
Does the consumer only need to access a specific API or Database? Private Link.

In an enterprise Landing Zone, logs are not just for troubleshooting; they are evidence. Log Archive vs. Monitoring Accounts

A common mistake in cloud architecture is treating logging and monitoring as the same task. While they use the same data, their Retention, Access, and Intent are fundamentally different.

1. The Log Archive Account (The Vault)

The Log Archive account is your Immutable Repository. Its primary purpose is long-term retention and audit compliance.

  • • Access: Extremely restricted. Typically, only the Audit/Compliance team has read access, and no one has delete access.
  • • Retention: Defined by legal requirements (e.g., 7 years).
  • • Storage: Optimized for cost using S3 Intelligent-Tiering or Glacier for older logs.

The Monitoring account (often where Security Hub lives) is your Real-Time Analysis center. It ingests data to trigger alerts and automated remediations.

  • • Access: Security Engineers and DevOps teams have access to view dashboards and logs for troubleshooting.
  • • Retention: Short-term (e.g., 30–90 days) for operational analysis.
  • • Storage: Higher performance (e.g., CloudWatch Logs, OpenSearch) to allow for fast querying and visualization.

Key Separation Rule:

If you delete an alert in the Monitoring account, the underlying log in the Log Archive account must remain untouched. Monitoring is for *now*; Log Archive is for *ever*.

Decommissioning & Data Retention

Closing an account is more than clicking a button. At the enterprise scale of Hexadigitall, it involves a multi-step orchestration to protect intellectual property and satisfy compliance audits.

1. The Offboarding Protocol

Before an account is formally closed in AWS Organizations, the following "Deep Clean" must occur:

  • • Resource Purge: Deleting all non-storage resources (EC2, RDS, Lambda) to stop billing immediately.
  • • Final Snapshot: Moving critical database snapshots and S3 data to the Log Archive or a dedicated "Cold Storage" account.
  • • Network Detachment: Removing Transit Gateway attachments and Peering connections to prevent traffic from hitting a void.

2. The Retention "Black Hole"

Legal requirements often mandate that even if an account is closed, its logs and primary data must exist for years.

  • • S3 Glacier Deep Archive: The destination for retired account data. Cost-effective for 7-10 year retention.
  • • Immutable Backups: Ensuring the final state of the account is captured in an AWS Backup vault with "Vault Lock" enabled.

Operational Risk Warning:

Never close a Management Account or an account that owns a Service Quota shared with the org. Always identify cross-account dependencies (KMS keys, Shared Subnets) before pulling the plug.

Workspaces vs. Per-Account Backends

The core challenge of managing 100+ accounts is State Collision. If multiple automated pipelines attempt to update different accounts that share a single state file or backend, the "Locking" mechanism will become a bottleneck.

1. The Workspace Approach (Centralized)

Workspaces allow you to use one set of configuration files to manage multiple distinct sets of infrastructure resources.

  • • Pros: Dry code; one single terraform init for all accounts.
  • • Cons: High risk. A "State rm" or "State push" error can accidentally target the wrong workspace. Performance degrades as the number of workspaces grows.

2. Isolated Backends (The Hexa-Standard)

In this model, every account (or small group of accounts) has its own S3 Key and DynamoDB Lock.

  • • Pros: Maximum blast-radius protection. Parallel execution across 100 accounts is physically isolated.
  • • Cons: Requires more complex automation (e.g., AFT or Terragrunt) to generate backend configurations dynamically.

The Architect's Verdict:

For Hexadigitall, we utilize Isolated Backends. While Workspaces are excellent for Dev/Stage/Prod within a single application, they lack the administrative separation required for a 100-account corporate Landing Zone.

BACK MATTER: REFERENCE
THE TRANSLATOR (GLOSSARY)

Technical Terms & Definitions

Governance & Structure

AWS Organizations: The foundational service for account consolidation. It enables hierarchical grouping into OUs and centralizes billing/policy enforcement.

Control Tower: An AWS managed service that orchestrates Organizations, SSO, and Config to set up and govern a secure multi-account environment.

OU (Organizational Unit): A logical container for accounts within an organization. Policies (SCPs) applied to an OU are inherited by all child accounts.

SCP (Service Control Policy): A JSON policy that sets the maximum available permissions for an account or OU. It acts as a "filter" and cannot grant permissions, only restrict them.

Blast Radius: The total impact surface area of a technical failure or security breach. Landing Zones use account boundaries to minimize this area.

Automation & State

AFT (Account Factory for Terraform): A GitOps-driven engine that automates account creation and the application of custom Terraform baselines.

Drift: When the "Actual State" of a cloud resource deviates from the "Desired State" defined in HCL code, usually due to manual console changes.

Idempotency: The property where an automation script (like Terraform) produces the exact same result whether run once or one thousand times.

State Locking: A mechanism using DynamoDB to prevent multiple concurrent Terraform executions from writing to the same .tfstate file and causing corruption.

Terragrunt: A thin wrapper for Terraform that provides extra tools for keeping configurations DRY (Don't Repeat Yourself) and managing multiple remote states.

Networking & Security

Transit Gateway (TGW): A regional network hub that acts as a cloud router, simplifying VPC peering by connecting thousands of VPCs and on-premises networks.

RAM (Resource Access Manager): A service used to share AWS resources (like TGWs or Subnets) across accounts within an organization.

PrivateLink: A technology that provides private connectivity between VPCs and AWS services without using public IP addresses or traversing the internet.

Guardrails: High-level rules. Preventative guardrails (SCPs) stop actions; Detective guardrails (AWS Config) notify you when a violation has already occurred.

IAM Identity Center (SSO): A centralized service to manage single sign-on access to all AWS accounts and cloud applications from a single Identity Provider.
